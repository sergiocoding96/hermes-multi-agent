MemOS: From Testing to Production

Continued from Session 1's MemOS infrastructure build. Focused on stress testing the search pipeline, enforcing real cube isolation, building per-agent auth, and replacing curl-based memory calls with a native Hermes plugin.

Search Recall Fix (1/12 → 5+ results)

Four compounding filters in the search pipeline were too aggressive. A vague query scored 0.10–0.25 similarity but got filtered at the 0.20 threshold before reaching any results.

Key Learning

Making search parameters env-configurable means quality tuning without code changes. Always make thresholds configurable when they affect retrieval quality.

Cube Isolation Fix (3/10 → 10/10)
Root Cause
UserManager.validate_user_cube_access() existed but was never called. Any user could read/write any cube by passing its ID. Trust-based only, no enforcement.
Fix Applied
Permission check added at the top of SearchHandler and AddHandler before any cube view is constructed. UserManager injected into HandlerDependencies.
Per-Agent API Key Authentication

Keys stored in agents-auth.json (ships with git repo). AgentAuthMiddleware validates Authorization: Bearer ak_... headers. Authenticated user_id stored in a ContextVar.

Key Insight: Trust vs Real Auth
With MEMOS_AUTH_REQUIRED=false, cube isolation only works if callers honestly identify themselves. Anyone can claim user_id: "ceo" and read everything. Setting auth to true closes this gap entirely. Auth middleware was built but disabled.

Native Tool Plugin: memos_store + memos_search

Agents were using curl commands from SKILL.md files. This leaked API keys into LLM context, wasted ~200 tokens per call, and was error-prone with malformed JSON.

Plugin: ~/.hermes/plugins/memos-toolset/
plugin.yaml — manifest, requires MEMOS_API_KEY, MEMOS_USER_ID, MEMOS_CUBE_ID
schemas.py — minimal tool schemas (agent sees content/query only)
handlers.py — HTTP calls with identity injected from env vars
__init__.py — registers tools via ctx.register_tool()
Key Design Decisions
✓ Identity from env, not LLM — agent never sees credentials
✓ Per-profile .env — each profile loads its own MEMOS vars
✓ No delete tool — LLMs shouldn't destroy memories
✓ LLM-formatted results — numbered list with rank + relevance
⚠️
MiniMax Key Bug Found
Profile .env files replace the global ~/.hermes/.env entirely. If a profile .env only has MEMOS vars, the MiniMax key is lost → 401. Fix: include MINIMAX_API_KEY and DEEPSEEK_API_KEY in every profile's .env.

Infrastructure was locked down. This session closed the remaining 8 application-level gaps: endpoint-level authorization for all routes, a working admin key management API, age-encrypted secrets, and full localhost binding.

The Auth vs Authorization Distinction
🔑
Auth gate ≠ Authorization
MEMOS_AUTH_REQUIRED=true only proves the caller has a valid key. It does NOT check which cubes they can access. Any authenticated agent could still read, write, or delete any other agent's data through ~10 unprotected endpoints.

Critical Fix: Cube ID Mismatch
Agents address cubes by user_id (e.g., "ceo"), but UserManager DB stores them as cube_id (e.g., "ceo-cube"). The original method only matched on cube_id, so every access check returned False. Fixed: method now resolves cubes by trying cube_id → cube_name → owner_id in sequence.

Blind Audit & Key Hardening

Structured 34-subtest blind audit against the now-hardened MemOS. Found 3 remaining issues: plaintext key storage, no rate limiting on auth failures, and static key registry. All fixed. Final score: 9.5/10.

Audit Results by Category
Authentication (10 subtests)8/10
Cube Isolation (9 subtests)10/10
Memory Quality (6 subtests)10/10
Edge Cases (9 subtests)10/10
Three Fixes Applied
1 · Bcrypt Key Hashing
agents-auth.json upgraded to v2 format. Raw keys printed once at provisioning, never stored. bcrypt.checkpw() validates incoming keys. Backward-compatible v1 support retained.
2 · Auth Failure Rate Limiting
In-memory sliding window per IP. 10 failures/60s → 429. Rate limit checked AFTER auth attempt — valid key from limited IP still succeeds and clears the counter.
3 · Auto-Reload Key Registry
_check_reload() compares mtime on every request (~0.01ms). Key rotation, adding agents, revocation — all without server restart.

Important Gotcha: Bcrypt Rate Limiting Order

The initial implementation checked rate limit BEFORE auth — this locked out legitimate users when an attacker triggered the limit from the same IP. Fix: attempt auth first. If successful, clear the failure counter. Only return 429 if auth also fails.

Paperclip Local Deployment & Recovery

Paperclip (CEO agent UI, Claude Opus 4.6) was unresponsive. The node process was alive but the embedded PostgreSQL subprocess had crashed. Diagnosed and recovered in one session.

Root Cause
Symptom: ECONNREFUSED 127.0.0.1:54329
Port 54329 is Paperclip's embedded PostgreSQL (@embedded-postgres/linux-x64). The DB subprocess had died (likely a system restart or OOM), but the node process kept running — alive but unable to serve any requests. Logs showed the error flooding every heartbeat tick.
Recovery Runbook
Recovery Commands

# 2. Restart with correct env (embedded postgres auto-starts)

export PAPERCLIP_AGENT_JWT_SECRET

# 3. Wait ~8s, then verify
curl http://localhost:3100/api/health

Key Technical Lessons

Patterns and insights that appeared across multiple sessions — the non-obvious things that would have taken much longer to discover without structured documentation.

Security
Never test your own security code
The builder passes tests the builder wrote. A zero-knowledge audit by a fresh session (no context, no hints) found 15 vulnerabilities that 3 rounds of self-testing completely missed. This is now a repeatable practice: after hardening, always run a blind audit with a clean session.
Auth ≠ Authorization — two separate layers
Having MEMOS_AUTH_REQUIRED=true only proves the caller has a valid key. It does nothing to restrict which cubes they can access. You need both: middleware-level authentication AND endpoint-level authorization checks on every route that touches data.
Cube ID != User ID — the #1 MemOS trap
Agents address cubes by user_id (e.g., "ceo"). The UserManager DB stores them as cube_id (e.g., "ceo-cube"). Any access check that queries by cube_id alone silently fails. Always resolve through owner_id as fallback. This caused every authorization check to return False until fixed.
Infrastructure
Native modules + Node upgrades
When Node.js major version updates (especially via Homebrew/Linuxbrew), native modules like better-sqlite3 must be rebuilt. Symptom: "Module did not self-register." Fix: npm rebuild <module>.
Paperclip zombie on DB crash
Paperclip's embedded postgres can die without killing the node process. The node stays alive but can't serve requests. ECONNREFUSED on port 54329 means: kill the node process and restart — don't try to revive postgres separately.
Qdrant host+port SSL bug
When api_key is set with host+port (not url), the Qdrant Python client defaults to HTTPS. Fix: explicitly pass client_kwargs["https"] = False for localhost connections in qdrant.py.
Profile .env replaces global .env
When a Hermes profile activates, its .env replaces the global ~/.hermes/.env entirely — not appends. Every profile .env must include MINIMAX_API_KEY and DEEPSEEK_API_KEY or the agent will 401 on model calls.

Architecture Decisions
Key Architectural Decisions
SearXNG over Brave Search
Free, unlimited, multi-engine. Brave credits exhausted in 6 days on agentic workloads. SearXNG's cross-engine scoring provides natural quality ranking (found by 3+ engines = high confidence). Returns 40+ results vs Brave's 5–10. Trade-off: occasional noise from engine disagreement, but agents filter this during synthesis.
Holographic + MemOS (not either/or)
Holographic = agent's personal learning (trust scoring, entities, contradictions, within-profile). MemOS = organizational knowledge (cross-agent, CEO reads all cubes). They serve different scopes and complement each other. Holographic for local reasoning, MemOS for shared state between agents.
Profile isolation + shared skills
Memory isolated (agents don't leak context to each other via separate MemCubes). Skills shared (improvements propagate via external_dirs + GitHub repo). Mirrors how human teams work: private notes, shared playbooks.
Collect trajectories without immediate RL
Zero cost to collect. High cost not to have when needed. Nightly trajectory export captures every session as potential training data for a future fine-tuned model that knows specific research methodology. Start now, fine-tune later.
Site-Packages Warning
⚠️
All MemOS patches live in site-packages
Every security fix, search tuning change, and auth improvement is in ~/.local/lib/python3.12/site-packages/memos/. A pip install --upgrade memos will silently overwrite all of them. These need to be upstreamed or maintained as a patch file. Do not upgrade MemOS without reapplying all patches.
