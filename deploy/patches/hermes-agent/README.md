# Local Hermes Agent patches

We install Hermes Agent from upstream (`NousResearch/hermes-agent`) via the
standard `curl install.sh | bash` flow — no fork, no rebase loop. Our local
fixes live here as numbered `.patch` files (mailbox format, generated by
`git format-patch`) and get applied on top after install.

## Why this layout

- **No public fork to maintain.** Upstream stays upstream.
- **Patches are versioned in the deploy repo**, so they survive an
  install wipe and travel with whoever clones `hermes-multi-agent`.
- **Catch-up is deliberate, not continuous.** When you want to sync with
  upstream, you do it as an explicit task: refresh the local install,
  re-apply patches, drop ones that are obsolete, refresh ones that
  rotted. No silent rebase surprises.

## Applying

```bash
bash deploy/scripts/apply-hermes-agent-patches.sh
```

The script is idempotent — patches already present in the local hermes-agent
git history (matched by commit subject) are skipped. Failures are loud and
tell you which file upstream likely moved under us.

## Adding a new patch

After committing a fix locally in `~/.hermes/hermes-agent/`:

```bash
cd ~/.hermes/hermes-agent
git format-patch -1 <commit-sha> \
    --output-directory=~/Coding/Hermes/deploy/patches/hermes-agent/
```

The script numbers patches by their leading `NNNN-` prefix and applies them
in lexical order. Use 4-digit prefixes (`0001-`, `0002-`, …) to leave room
for inserts.

## Catching up with upstream — the periodic task

Roughly: pull a fresh upstream, re-apply patches, triage the failures.

```bash
# 1. Fresh upstream
cd ~/.hermes/hermes-agent
git fetch origin
git checkout main
git reset --hard origin/main          # WARNING: drops local commits

# 2. Re-apply our patches
bash ~/Coding/Hermes/deploy/scripts/apply-hermes-agent-patches.sh
```

For each patch that fails to apply:

1. **Read the failing file in upstream.** Did upstream already fix the
   underlying bug? If yes, **delete the patch** — your fix is now obsolete.
2. **If upstream just refactored around it,** re-create the patch against
   the new code:
   ```bash
   cd ~/.hermes/hermes-agent
   # ... manually re-apply the fix to the new upstream code, commit it
   git format-patch -1 <new-sha> --output-directory=<patches-dir>/
   # delete the old patch file
   ```
3. **Commit the catch-up to hermes-multi-agent** (`chore(patches):
   refresh after upstream sync`) so the patches dir reflects the new
   upstream baseline.

## What's currently here

- `0001-fix-gateway-clear-platform-error-fields-when-caller-.patch` —
  fixes `gateway/status.py` so platform error fields actually clear when
  the caller passes `None` (e.g. `_mark_connected()` after recovering
  from a transient telegram disconnect). See the patch's own commit
  message for full root cause.

- `0002-feat-state-add-secret-redactor-module-for-state.db-h.patch` —
  introduces `hermes_state_redactor.py` (regex set + Luhn-validated card
  detection + substring fast-path). Standalone module — no behavioral
  change yet; subsequent patches wire it in. Patterns mirror MemOS
  PR #6's redactor.

- `0003-fix-state-redact-secrets-on-write-into-messages-tabl.patch` —
  PR-A. Hooks the redactor into `add_message` and `replace_messages` so
  secrets in `content` and `tool_calls` get scrubbed before they're
  INSERTed into `state.db`. The FTS index populated by after-INSERT
  triggers also indexes only the redacted form.

- `0004-fix-state-redact-secrets-on-context-load-reads-PR-B.patch` —
  PR-B. Hooks the redactor into `get_messages_as_conversation` so loaded
  message history is scrubbed *before* it enters the LLM's prompt context.
  Higher-leverage than PR-A because it stops the recurring cross-turn
  leak even when `state.db` rows contain raw secrets from before PR-A
  landed (e.g. historical data on existing deployments).

- `0005-fix-state-redact-secrets-on-memory-tool-writes-reads.patch` —
  PR-C. Closes the second persistence channel discovered during PR #26
  deploy verification: `tools/memory_tool.py` writes
  `memories/USER.md` and `memories/MEMORY.md`, both of which are loaded
  verbatim into the system prompt at session start. Hooks `redact()`
  into `add()` and `replace()` (write-side) and `_read_file()`
  (read-side defense for pre-patch raw entries — self-heals on next
  rewrite, mirrors PR-B's pattern).

- `0006-fix-state-redact-secrets-on-session-log-JSON-writes-.patch` —
  PR-D. Closes the third persistence channel: `sessions/session_<id>.json`
  dumps written by `_save_session_log()` in `run_agent.py`. Not a leak
  *into* LLM context (session_search uses SQLite) but a leak *out* via
  bug reports / `hermes dump` / profile backups. Walks every string
  field in the entry via `redact_dict()`.

  **Operator note:** for existing `state.db`, `memories/USER.md`,
  `memories/MEMORY.md`, and `sessions/*.json` files with un-redacted
  historical content, run the one-shot scrubber at
  `deploy/scripts/scrub-hermes-state-secrets.py`. It auto-discovers all
  four file types under a profile root::

      python3.12 deploy/scripts/scrub-hermes-state-secrets.py \
          --profile ~/.hermes/profiles/research-agent \
          --profile ~/.hermes/profiles/email-marketing

  The patches handle the forward path; the scrubber handles legacy data.

  Closes hermes-multi-agent issue #24 (T2 from the v1 smoke test).

## Known wart in the apply script

`apply-hermes-agent-patches.sh` checks idempotency by `git cat-file -e
<sha-from-patch-From-line>`. That works for patches whose `From:` SHA
matches a local commit (e.g. `0001` references an upstream commit
already in `origin/main`), but **not** for patches applied via `git am`
on a different machine — `git am` rewrites the SHA on apply. Result:
on second-run, patches `0002`–`0006` all attempt to re-apply and fail
loudly. The fix is obvious (match by commit subject + tree-state, not
SHA) but is out of scope for this PR; track it separately.
